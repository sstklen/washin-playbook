# Cron Generated Script Desync

## Problem

A setup script generates a runtime script (e.g., `~/check-and-deploy.sh`) using a
heredoc (`cat > ~/script.sh << 'EOF' ... EOF`). A cron job runs this generated copy.
When bugs are found and fixed in the SETUP script in the repo, `git pull` updates
the setup script but NOT the generated copy. The cron keeps running the old buggy version.

This creates a maddening loop where:
1. Bug is found (e.g., every-minute spam notifications)
2. Fix is committed to the repo's setup script
3. `git pull` on server updates the repo
4. But the RUNNING script (`~/script.sh`) is unchanged
5. Bug persists despite "fix" being deployed
6. Developer tries again, same result -> "why won't this fix work?!"

## Context / Trigger Conditions

- A cron job runs a script that lives OUTSIDE the git repo (e.g., `~/check-and-deploy.sh`)
- The script was originally generated by a setup script using heredoc
- Fixing the setup script in the repo doesn't fix the running behavior
- Multiple fix attempts all fail with the same symptoms
- The "fix" works locally but not on the server

**Key diagnostic question**: "Is the cron running a script FROM the repo, or a COPY?"

```bash
# Check: what path does cron actually execute?
crontab -l | grep -v "^#"
# If path is ~/script.sh or ${HOME}/script.sh -> it's a COPY (bad)
# If path is ~/project/scripts/script.sh -> it's from repo (good)
```

## Solution

### Architecture Fix (Permanent)

Move the script logic INTO the repo and have cron call it directly:

```bash
# BEFORE (broken pattern):
# setup.sh generates ~/check-and-deploy.sh via heredoc
# cron: * * * * * ~/check-and-deploy.sh
# -> git pull does NOT update ~/check-and-deploy.sh

# AFTER (correct pattern):
# Script lives at ~/project/scripts/check-and-deploy.sh (in repo)
# cron: * * * * * ~/project/scripts/check-and-deploy.sh
# -> git pull DOES update the script automatically
```

**Steps:**
1. Create the script directly in the repo (e.g., `scripts/check-and-deploy.sh`)
2. Update the setup script to:
   - Point cron to the repo path instead of generating a copy
   - Delete the old generated copy if it exists
   - `chmod +x` the repo script
3. On the server, run the updated setup script once
4. Verify cron now points to the repo path

### Safety Net: Notification Cooldown

Even with the architecture fix, add a notification cooldown to prevent spam
if the logic ever has bugs again:

```bash
LAST_NOTIFY_FILE="${HOME}/.last-notify-timestamp"
COOLDOWN_SEC=1800  # 30 minutes

notify() {
    local msg_type="${2:-default}"
    local cooldown_key="${LAST_NOTIFY_FILE}.${msg_type}"

    if [ -f "$cooldown_key" ]; then
        local last=$(cat "$cooldown_key" 2>/dev/null || echo 0)
        local elapsed=$(( $(date +%s) - last ))
        if [ "$elapsed" -lt "$COOLDOWN_SEC" ]; then
            return 0  # Still in cooldown
        fi
    fi

    # Send notification...
    curl -sf -X POST "..." || true

    # Record timestamp
    date +%s > "$cooldown_key" 2>/dev/null || true
}
```

### Quick Fix (If You Can't Restructure)

If moving the script to the repo isn't feasible, add self-update logic:

```bash
#!/bin/bash
# At the top of the generated script, add:
REPO_VERSION="~/project/scripts/setup.sh"
MY_PATH="$0"

# Check if repo version is newer -> regenerate
if [ "$REPO_VERSION" -nt "$MY_PATH" ]; then
    bash "$REPO_VERSION"  # Re-run setup to regenerate this script
    exec "$MY_PATH" "$@"  # Re-exec with new version
fi
```

## Verification

```bash
# 1. Check cron points to repo path
crontab -l | grep -v "^#"
# Should show: ~/project/scripts/check-and-deploy.sh (NOT ~/check-and-deploy.sh)

# 2. Verify old copy is removed
ls ~/check-and-deploy.sh 2>/dev/null && echo "OLD COPY STILL EXISTS!" || echo "Clean"

# 3. Wait a few minutes and check no spam notifications
# Monitor Telegram/Slack/email for unwanted messages
```

## Example

Real-world case from Washin Village API auto-deploy:

- **Symptom**: Telegram bot sending "deploy success + update content" every 1 minute
- **Attempted fixes** (both failed):
  1. Added hash tracking to `auto-deploy-setup.sh` in repo
  2. Added `git checkout -- .` before pull in `auto-deploy-setup.sh`
- **Why fixes failed**: `auto-deploy-setup.sh` GENERATES `~/check-and-deploy.sh` via
  heredoc. The cron runs `~/check-and-deploy.sh` (the copy), not the repo version.
  `git pull` updates the setup script but never regenerates the copy.
- **Root fix**: Put deploy logic directly in `scripts/check-and-deploy.sh` in repo,
  cron calls repo path, setup script deletes old copy.

## Notes

- This anti-pattern is common in "one-time setup" scripts that generate runtime configs
- Same issue applies to systemd service files, Docker entrypoints, and any config
  that's GENERATED from a template but then lives independently
- Bash does NOT necessarily read the entire script before executing, so modifying a
  running script via git pull could theoretically cause issues (but in practice, for
  small scripts under 4KB, the kernel buffers it all)
- Always add notification cooldown as defense-in-depth, regardless of fix correctness
