# AI Agent 審計報告假問題防治

## Problem

當派 Sub-agent 去做「找問題」「盤點缺口」「審計」類任務時，回報的問題清單
有嚴重的膨脹偏差。直接呈報用戶會損害信任，因為大多數「問題」經驗證後是假的。

**2026-02-13 實證案例：**
- 派 Explore agent 審計用戶旅程，回報 18 個「關鍵缺口」
- 用戶要求逐條驗證
- 結果：只有 1 個是真問題（回報系統確實不存在），其他 17 個都已在代碼中處理
- 真實率：5.5%

## Context / Trigger Conditions

**觸發場景（只要產出問題清單就要啟用）：**
- 盤點（「幫我檢查整個系統」「目前狀況如何」）
- 審計（「找出所有問題」「安全檢查」「品質審查」）
- Gap 分析（「還缺什麼」「哪裡沒做好」）
- Code Review（「審查這段代碼」「有沒有 bug」）
- UX 審計（「用戶流程有問題嗎」「哪裡會卡住」）

**膨脹偏差的三個根因：**

### 根因 1：激勵錯位
派 agent 去「找問題」，agent 的隱性激勵是找越多越好，顯得越全面越專業。
沒有「假陽性成本」的概念 — 列出假問題不會被懲罰，漏列真問題才會。

### 根因 2：單層分析
Agent 只看當前代碼片段，不追蹤完整流程。
- 看到 Landing Page 沒有登入按鈕 → 列為問題
- 但沒追下去看 Landing → /beta → /beta 有登入按鈕（一步到位）

### 根因 3：缺乏反向驗證
找到疑似問題後，沒有回去確認代碼是否已處理。
- 看到「token balance不足」場景 → 列為「死胡同」
- 但沒查 402 回應裡已有 `howToEarnTokens` 引導欄位

## Solution

### 強制驗證流程（Mandatory Verification Protocol）

```
Sub-agent 回報問題清單
        ↓
┌─────────────────────────────┐
│  主 Agent 逐條驗證          │
│                             │
│  對每個問題：               │
│  1. Grep/Read 找相關代碼    │
│  2. 問：代碼已經處理了嗎？   │
│  3. 追蹤完整流程（不只看一層）│
│  4. 標記 [已驗證✅] 或 [假❌] │
└─────────────────────────────┘
        ↓
只呈報 [已驗證✅] 的問題
附代碼行號或測試結果作為證據
```

### 呈報格式要求

**禁止：**
```
## 發現 18 個問題
1. Landing Page 沒有登入按鈕
2. token balance不足時的死胡同
3. ...
```

**正確：**
```
## 驗證結果：18 項中 1 項為真實問題

### ✅ 已驗證的問題（1 項）
1. 沒有 Bug 回報管道
   證據：全站 Grep "feedback|bug|report" 零結果

### ❌ 排除的假問題（17 項）
1. "Landing 沒登入按鈕" → /beta 第 1669 行有登入 CTA
2. "token balance不足死胡同" → api-proxy.ts 第 1573 行有 howToEarnTokens
...
```

### 反向驗證 Checklist

對每個疑似問題，執行以下檢查：

| 步驟 | 動作 | 工具 |
|------|------|------|
| 1 | 搜尋相關關鍵字 | `Grep` |
| 2 | 讀取相關代碼段 | `Read` |
| 3 | 追蹤流程下一步 | `Read`（跟著 link/redirect） |
| 4 | 實際測試端點 | `Bash`（curl） |
| 5 | 標記結果 | ✅ 真 / ❌ 假 / ⏳ 待確認 |

### 給 Sub-agent 的 Prompt 改進

**之前（會膨脹）：**
```
找出所有用戶旅程的缺口和問題
```

**改進（限制膨脹）：**
```
分析用戶旅程。對每個發現的疑似問題，你必須：
1. 附上具體的代碼行號
2. 確認代碼中沒有已存在的處理邏輯
3. 區分「確認存在的問題」和「可能的改善建議」
不要列出你沒有代碼證據的問題。
```

## Verification

驗證此 skill 有效的標準：
- 呈報給用戶的問題清單中，真實率 > 80%（之前是 5.5%）
- 每個問題都附有代碼證據（行號或測試結果）
- 用戶不再說「先查查是真的假的」

## Example

**場景：** 用戶要求「幫我檢查整個系統還缺什麼」

**步驟 1：** 派 Explore agent 掃描
```
Agent 回報：發現 8 個問題
1. 缺少 404 錯誤頁面
2. 沒有 rate limiting
3. API 回應沒有 CORS header
...
```

**步驟 2：** 主 Agent 逐條驗證
```bash
# 問題 1：缺少 404 錯誤頁面
Grep "404|notFound|not.found" → 找到 http-server.ts:3500 有自訂 404 頁
→ ❌ 假問題

# 問題 2：沒有 rate limiting
Grep "rateLimit|rate.limit|rateLimiter" → 找到 middleware 有 100/min 限制
→ ❌ 假問題

# 問題 3：API 回應沒有 CORS header
curl -I https://api.example.com → 確認沒有 Access-Control 標頭
→ ✅ 真問題（附 curl 結果為證據）
```

**步驟 3：** 呈報
```
驗證結果：8 項中 1 項為真實問題

✅ API 回應缺少 CORS header
   證據：curl -I 回應中無 Access-Control-Allow-Origin

其他 7 項經驗證均已在代碼中處理。
```

## Notes

- 這個偏差在「找問題」「找缺口」類任務中最嚴重，在「實作」「修復」類任務中較少
- Sub-agent 的分析深度通常只有 1-2 層，複雜的多步驟流程容易誤判
- 問題清單越長，假陽性率通常越高（因為真問題先找到，後面的越來越勉強）
- 即使是人類 code review，假陽性率也常見，但 AI agent 因為沒有「不確定」的表達習慣，偏差更嚴重
- See also: `api-security-audit-methodology`（安全審計有類似的多輪驗證策略）
