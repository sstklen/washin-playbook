# Token Economics Audit Methodology

## Problem

Token/積分經濟系統設計完後，往往看起來合理但隱藏致命漏洞。特別是：
- 多層加成疊加後，互刷變得有利可圖
- 基金收支不平衡導致結構性破產
- 防禦機制太嚴格反而趕走好用戶
- 沒有反自動化，被 bot 批量套利

這些問題在單純閱讀規格書時很難發現，需要系統化的紅隊思維。

## Context / Trigger Conditions

- 完成 token 經濟系統設計稿 v1.0+
- 系統包含：賺取（mint）、消費（burn）、等級加成、被動收入
- 有多方參與者（消費者、貢獻者、鯨魚、新人）
- 需要在開發前找出致命設計缺陷

## Solution

### multi-agent joint audit架構

同時派出三個獨立審計角色，各有不同視角：

```
審計架構：
├── 紅隊（Red Team）— 攻擊者思維
│   目標：找出所有可被 gaming 的漏洞
│   方法：假設自己是理性攻擊者，計算套利空間
│   輸出：漏洞清單 + 嚴重度 + 攻擊成本vs收益
│
├── 轉化設計審計（Conversion Audit）— 行為引導思維
│   目標：驗證每個防禦機制是否兼顧「引導轉化」
│   方法：逐一檢查每個機制的「阻止型」vs「引導型」比例
│   輸出：轉化設計評分 + 改善建議
│
└── 經濟模型驗算（Economic Validation）— 數學思維
    目標：驗證數字平衡、基金永續性
    方法：模擬極端場景（全是鯨魚、全是新人、大量互刷）
    輸出：平衡計算 + 破產風險分析
```

### 紅隊攻擊清單（必查項目）

| # | 攻擊向量 | 檢查什麼 | 致命標準 |
|---|---------|---------|---------|
| 1 | **互刷套利** | A 幫 B 用 Key，B 幫 A 用 Key，雙方淨收益 > 0？ | 每輪淨利 > 0 = FATAL |
| 2 | **等級加成疊加** | 百分比加成 x 加速乘數 = 實際倍率是多少？ | 倍率 > 1.2x = FATAL |
| 3 | **基金永續性** | 收入（手續費%）vs 支出（加成%），誰大？ | 支出 > 收入 = FATAL |
| 4 | **Sybil 攻擊** | 建 10 個假帳號互相推薦，總收益？ | 有利可圖 = HIGH |
| 5 | **被動收入上限** | 不操作也能賺多少？有沒有天花板？ | 無上限 = HIGH |
| 6 | **服務商灌 Key** | 註冊商業帳號灌入 100 把 Key？ | 無速率限制 = HIGH |
| 7 | **自動化腳本** | 寫 bot 自動貢獻+消費循環？ | 無 CAPTCHA/OTP = MEDIUM |

### 互刷套利數學驗證公式

```
// 最重要的驗算！
scenario: A 和 B 互相用對方的 Key

A 的成本 = 消費 WT（用 B 的 Key）
A 的收益 = 貢獻獎勵（Key 被 B 用）+ 等級加成 + 加速加成
A 的淨收益 = 收益 - 成本 - 手續費

如果 A_淨收益 + B_淨收益 > 0 → FATAL！互刷有利可圖

修復公式：
finalEarning = min(
  baseEarning * accelerationRate,
  consumerSpend * 0.85  // 收益安全閥：永遠不超過消費方支出的 85%
)
```

### 轉化設計評分標準

每個防禦/激勵機制都評為以下三類之一：

| 類型 | 定義 | 理想比例 |
|------|------|---------|
| **引導型** | 讓壞路成本 > 好路收益，自然轉化 | 80%+ |
| **混合型** | 有阻止也有引導，可改善 | 15% |
| **阻止型** | 純粹 block，沒有替代好路 | < 5% |

**改善策略：** 把「阻止型」改成「引導型」

- 429 硬擋 → 溢出付費（+50% WT，但可以繼續用）
- 72h 觀察期 → 行為驗證門（做好事可以提前解鎖）
- 固定手續費 → 信任折扣（Trust 越高費率越低）

### 修復模式庫

| 漏洞 | 修復模式 | 關鍵參數 |
|------|---------|---------|
| 百分比加成互刷 | 改為**固定每日金額** | 不跟交易量掛鉤 |
| 基金破產 | **月預算制**（上月收入=本月預算） | 不足時按比例縮減 |
| 無上限被動收入 | **收益安全閥** | finalEarning <= consumerSpend * 0.85 |
| 加速倍率太高 | **硬上限 1.10x** | 分 4 級逐步遞增 |
| 觀察期趕走好人 | **行為驗證門**（3 條解鎖路徑） | 72h 兜底 |
| 鯨魚無動力 | **信任費率折扣**（15%→8%） | 按 Trust 分 4 級 |

## Verification

審計完成後，必須做以下驗證：

1. **互刷驗算**：用最優攻擊場景重新計算，確認淨收益 ≤ 0
2. **基金平衡**：模擬 1000 用戶場景，基金月收支是否平衡
3. **轉化評分**：所有機制重新評分，目標 100% 引導型
4. **Grep 驗證**：搜尋整份文件，確認舊的百分比加成已全部替換

## Example

### 實際案例：API Platform v1.3 → v1.4

**發現的致命漏洞：**

1. **FATAL - 等級加成 x Key 加速 = 互刷印鈔機**
   - v1.3：等級加成 +30%（百分比）x Key 加速 1.25x = 1.625x
   - 互刷每輪淨利 +0.38 WT
   - v1.4 修復：等級加成改固定 +30 WT/日 + 加速上限 1.10x + 收益安全閥

2. **FATAL - 社群基金結構性破產**
   - v1.3：收入 0.05 WT/tx（5%手續費），支出 0.255 WT/tx（30%加成）
   - 比例 1:5，必定破產
   - v1.4 修復：月預算制，上月 5% 收入 = 本月預算

3. **HIGH - 無反自動化**
   - v1.4 修復：Email OTP + IP 限制 + Key 貢獻速率限制

**結果：轉化設計評分從 80%（12/15 引導型）提升到 100%（15/15 引導型）**

## Notes

- 審計時機：規格書完成後、開發前，此時修改成本最低
- 三個審計角色必須**獨立運行**（不共享發現），最後再整合
- 「引導型」不等於「不嚴格」，而是嚴格但提供替代好路
- 收益安全閥是最重要的防線，必須硬編碼在最底層
- 這個方法論適用於任何積分/token/信任經濟系統，不限區塊鏈
